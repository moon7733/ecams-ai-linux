이전 작업 연결고리. [`async-jobs/`](../async-jobs/) 결정 13~16 (PWA 강제종료 자동 복구) 완성 후 시작. 본 작업은 그 위에 "사용자가 다시 안 들어와도 알림으로 끌어들이기" 를 얹음.

작업 시작일. 2026-06-04.

## 사용자 결정 대기 (4가지)

본격 코딩 진입 전 사용자 확정 필요.

### 미결정 1. 권한 요청 타이밍

- (a) 첫 PWA 설치 감지 직후 (자동) — 사용자가 PWA 의 가치 확신 전이라 거부율 높음
- (b) **첫 질문 보낼 때** (sendMessage 진입 직후) — 알림이 왜 필요한지 사용자도 직관적. **추천**
- (c) 사이드바의 토글 버튼 (사용자 능동)
- (d) 첫 답변 완료 직후 "다음부터는 알림으로 알려드릴까요?"

### 미결정 2. 푸시 보낼 조건

- (a) 항상 답변 완료 시 (사용자가 화면 보고있어도)
- (b) document.hidden 일 때만 — 클라이언트가 판단 후 알림 표시
- (c) **서버측 `subscribers.size === 0` 일 때만 전송** — 서버가 SSE 구독자 유무로 판단. 가장 효율적, 비용 최소. **추천**

### 미결정 3. 알림 내용

- 제목 옵션. (a) "eCAMS AI 답변 완료" 고정 / (b) **질문 첫 40자 — 추천**
- 본문 옵션. (a) **"답변이 완료되었습니다" 고정 — 추천** / (b) 답변 첫 100자 (페이로드 크기 + 민감 정보 노출 위험)
- 클릭 시. **PWA 열림 + 해당 chat 자동 표시** (`?resumeJob=<jobId>` 쿼리)

### 미결정 4. 권한 거부 후 재요청 cooldown

- (a) 영구 (다시 권유 안 함)
- (b) **24시간 cooldown — 추천**. 사용자 마음 바뀔 수 있음
- (c) 7일 cooldown
- (d) 매 세션마다 재시도 (스팸)

## 결정 1. iOS Web Push 의 제약 사항 명시 — 2026-06-04

**배경**. iOS 16.4+ 부터 Web Push 지원. 단 조건. (1) Safari 만 (Chrome iOS 안됨), (2) **홈 화면에 추가한 PWA 만** (Safari 탭에선 안 옴), (3) 첫 알림 도착까지 지연 큼 (수십초~분), (4) 저전력 모드면 더 늦거나 안 옴.
**선택**. v1 에선 "iOS 는 best-effort" 명시. 안드 (FCM 사용, 즉시) 가 메인 경로. iOS 도 동작은 시키되 사용자 가이드에 "iOS 는 가끔 알림 늦을 수 있음" 안내.
**파급**. 결정 13 의 init resume 가 그래도 답변을 사라지지 않게 해줌. 알림이 늦더라도 PWA 다시 열면 답변 있음 → iOS 사용자도 큰 손해는 없음.

## 결정 2. VAPID 키 — 첫 기동 시 자동 생성 + JSON 영속화 — 2026-06-04

**선택**. 첫 server 기동 시 `.vapid.json` 없으면 `web-push.generateVAPIDKeys()` 호출 → publicKey/privateKey 저장. 이후 기동마다 같은 파일 로드.
**대안 (탈락)**. 환경변수 (`VAPID_PUBLIC_KEY`/`VAPID_PRIVATE_KEY`) 로 관리. 로컬 PC 운영이라 .env 관리 부담만 추가. 자동 생성이 단순.
**위험**. `.vapid.json` 분실 시 모든 기존 subscription 무효 — 사용자 재구독 필요. 백업 권고. 사내 운영이라 ROI 충분.

## 결정 3. Subscription 저장 — JSON 파일 영속화 — 2026-06-04

**선택**. `pushSubscriptions.json` 에 `{ userId, endpoint, keys: { p256dh, auth }, createdAt, lastSeenAt }` 배열로. 메모리 Map + write-through.
**대안 (탈락 1)**. 메모리만 — 서버 재기동마다 모든 알림 끊김. 불가.
**대안 (탈락 2)**. SQLite — 의존성 추가. 본 프로젝트는 JSON 영속화 패턴 (chatHistory localStorage, answer_cache.json 등) 으로 일관됨.
**이유**. JSON 이 기존 컨벤션과 일치. subscription 수십~수백개 규모면 충분.

## 결정 4. 푸시 전송 조건 — 서버 측 `subscribers.size === 0` 게이팅 (미결정 2 추천) — 2026-06-04

**시점**. `runChatJob` 의 답변 완료 직후 ( `finishJob` 호출 직전 또는 직후).
**조건**. `jobs[jobId].subscribers.size === 0` — 즉 사용자가 현재 SSE 구독 중이 아님 = 화면 안 보고 있음.
**파급**. 화면 보는 사용자에겐 노이즈 0. 백그라운드/PWA 종료 사용자에겐 정확히 도달.
**탈락 대안**. 클라이언트가 document.hidden 으로 판단 — 서버에서 한 단계 더 멀어짐, 페이로드 낭비.

## 결정 5. 권한 거부 후 — 영구 차단 + 사이드바 알림 토글로 재활성화 (사용자 확정) — 2026-06-04

**선택**.
- 권한 모달은 첫 질문 시 1회만. denied 후엔 모달 _다시 안 띄움_ (사용자 명시적 거부 존중).
- 대신 **사이드바 하단에 항상 보이는 🔔 알림 토글** 배치 — 현재 상태 표시 ("알림 켜짐 / 꺼짐 / 차단됨"). 사용자가 켜면 권한 요청, 끄면 unsubscribe.
- 브라우저 권한이 `denied` 상태에서 사용자가 토글 켜기 누르면 → "브라우저 설정에서 알림 허용 후 다시 시도해주세요" 안내 모달 (path 안내).
**이유**. (a) "다시 권유" 가 마치 스팸처럼 느껴질 수 있음 → 한 번 거부 = 모달 종료 가 깔끔. (b) 그러나 사용자 마음 바뀔 수 있어야 함 → 항상 보이는 토글이 능동적 재활성화 통로. (c) 사이드바 토글은 PWA/데스크탑 모두 일관.

## 결정 9. 권한 요청 타이밍 — 첫 질문 보낼 때 (사용자 확정) — 2026-06-04

**선택**. `sendMessage` 진입 직후, `Notification.permission === 'default'` 인 경우만 모달 표시. 1초 후 (질문 전송 직후 답변 대기 화면 위에) 자연스럽게 뜨도록.
**모달 내용**. "답변 완료 시 알림을 보내드릴까요? 다른 앱을 보고 있어도 답변이 끝나면 알려드립니다." + [예 / 다음에]. 예 누르면 `requestPermission` → granted 면 subscribe 흐름.
**이유**. 사용자가 _왜_ 알림이 필요한지 가장 직관적인 시점. 권한 다이얼로그 거부율 최소화.

## 결정 10. 알림 페이로드 — 제목=질문 첫 40자, 본문=고정 (사용자 확정) — 2026-06-04

**선택**.
- 제목. `originalText.slice(0, 40) + (length > 40 ? '...' : '')`. 빈 문자열이면 "사진 분석".
- 본문. 고정 `"답변이 완료되었습니다. 탭하여 확인하세요."`.
- 아이콘. `/icons/icon-192.png`. badge 동일.
- data. `{ type: 'job-complete', jobId, chatId }`.
- tag. `jobId` (결정 7 의 중복 방지).
**이유**. 답변 내용 자체는 페이로드에 안 실음 — 4KB 제한 + 민감 정보 노출 회피. 사용자는 제목으로 "어느 질문" 인지 식별 가능.

## 결정 6. 알림 클릭 시 PWA 열기 — `?resumeJob=<jobId>` 쿼리 + window message — 2026-06-04

**시나리오**. (a) PWA 가 이미 열려있음. (b) PWA 가 닫혀있음.
**선택**.
- (a) SW 의 `notificationclick` 에서 `clients.matchAll({ type: 'window' })` → 있으면 첫 윈도우 focus + `postMessage({ resumeJob: jobId })`. 클라이언트는 window 의 message 리스너에서 받음.
- (b) `clients.openWindow('/?resumeJob=' + jobId)` 로 새 윈도우. `init()` 의 activeJob 복구 분기에서 쿼리 우선 처리.
**파급**. 결정 13 의 init resume 로직과 자연스럽게 통합. SW 단에서 직접 DOM 조작 못 하므로 message + 클라이언트 로직 조합 필수.

## 결정 7. tag: jobId 로 알림 중복 방지 — 2026-06-04

**시나리오**. 같은 job 에 대해 서버가 푸시 두 번 보냈거나, 알림 도착 후 SW 가 재실행되며 또 표시 시도.
**선택**. `showNotification(title, { tag: jobId, ... })`. 같은 tag 의 기존 알림은 새 것으로 _대체_ 됨 (OS 알림 트레이에 중복 안 쌓임).

## 결정 8. iOS-only 안내 메시지 (v2) — 2026-06-04

**시나리오**. iOS Safari 탭으로 접속한 사용자가 권한 요청 모달에서 거부하면 cooldown 적용. 그러나 그들에게 진짜로 필요한 안내는 "홈 화면에 추가해야 알림 받을 수 있음" 임.
**선택**. v1 에선 [`mobile-pwa/`](../mobile-pwa/) 의 install 배너가 이미 iOS 안내 ("공유 → 홈 화면에 추가") 제공. 별도 push-specific 안내는 v2 로 미룸.
