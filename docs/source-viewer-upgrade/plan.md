# 소스뷰어 업그레이드 — Plan

## 목표
소스뷰어에 4가지 개선을 적용한다.

1. **모바일 가상키보드 억제** — CodeMirror read-only 뷰어를 터치하면 입력패드가 뜨는 불편 제거. 편집 불가인데 키보드만 열림.
2. **AI 답변 파일칩의 라인범위 처리** — AI가 `CheckOutRequestPage.java#L155-L172` 형태로 라인을 명시하면, 칩이 `...java#L155-L172` 경로로 파일을 찾아 실패함. `#L<start>-L<end>` 프래그먼트를 분리해 파일을 열고, 해당 라인범위를 선택+배경하이라이트로 "티나게" 표시.
3. **파일 단위 AI 분석** — "소스 다운로드" 옆 "소스분석" 버튼. 현재 파일을 LLM에 보내 분석 결과 수신. 분석 중에는 "분석 중입니다" 플로팅 표시. 결과는 데스크탑 = `소스 | 분석결과` 좌우 분할, 모바일 = `소스 / 분석결과` 상하 분할.
4. **드래그 영역 AI 질문** — 코드 드래그 선택 후 데스크탑은 우클릭, 모바일은 선택 직후 "드래그한 영역 질문" 버튼이 떠서 선택 영역만 분석. 표시는 3번과 동일.

## 아키텍처 결정
- **분석 LLM**: 앱 주력인 Claude Code CLI 재사용 (이미 인증됨, 추가 키 불필요). RAG용 `getSystemPrompt()` 는 사용하지 않고, 단일파일 분석 전용 lean 스폰(MCP off, max-turns 1, 도구 금지).
- **엔드포인트 통합**: `POST /api/fs/analyze` 하나로 3·4 모두 처리. body = `{ text, filename, question? }`. 클라이언트가 이미 디코드한 텍스트(또는 선택영역)를 전송 → 서버 파일 재read 불필요(경로이탈 위험 0, native2ascii 디코드된 한글이 그대로 LLM 입력).
- **입력 크기 상한**: 텍스트 80KB 초과 시 거부/절단(LLM 비용·지연 가드). 선택영역(4번)은 거의 안 걸림.
- **비스트리밍**: 스피너 + 단일 응답. Express `server.timeout=0` 이라 30~60s 응답 안전. 요구사항이 스트리밍을 요구하지 않음.
- **결과 비영속(ephemeral)**: 세션 한정. 새로고침/히스토리 저장 안 함.

## 검증
- 번들 변경(1·2·4) → `npm run build:cm`.
- server.js 변경(3·4) → pm2 재시작. API가 JSON 대신 `<`(HTML) 반환이면 서버 미재시작 신호.
