# Codex + Claude Code 협업 운영 계획

## 목적

eCAMS AI 개발을 최우선 목표로 두고, Codex와 Claude Code가 같은 레포에서 충돌 없이 이어받아 개발하도록 운영 규칙을 정한다.

## 기본 원칙

- eCAMS AI 기능 정확도, 유지보수성, 검증 가능성을 최우선으로 둔다.
- Codex와 Claude Code는 각자 작업 전 `MEMO.md`, 관련 `docs/<feature>/checklist.md`, `context-notes.md`를 먼저 읽는다.
- 새 기능이나 비단순 수정은 항상 `docs/<feature>/plan.md`, `checklist.md`, `context-notes.md`를 만든 뒤 시작한다.
- 문제가 생기면 바로 코딩하지 않고 원인 가설, 검증 방법, 수정안을 문서화한 뒤 서로 검토한다.
- 합의가 필요한 변경은 한쪽이 구현하고 다른 쪽이 리뷰하는 방식으로 진행한다.
- 유료 API 키 기반 모델 호출은 도입하지 않는다. 무료 Gemini 키는 되묻기와 임베딩 보조에만 사용하고, GPT 계열은 월구독 세션 기반 exec 방식만 후보로 둔다.

## 역할 분담

### Codex가 주로 맡기 좋은 일

- 현재 로컬 파일 구조 파악, `rg`, 테스트, 빌드, Git 상태 확인.
- Windows PowerShell 기반 실행과 Codex 앱의 diff, 터미널, 브라우저 확인.
- 작은 범위의 수술적 수정.
- 문서화, 체크리스트 갱신, 작업 로그 정리.
- 반복 조사에서 `cavecrew`를 사용해 토큰 소모를 줄이는 작업.

### Claude Code Opus가 주로 맡기 좋은 일

- 복잡한 설계 판단, 장기 아키텍처 방향, 큰 리팩터링 전략.
- 애매한 요구사항의 해석과 대안 비교.
- 깊은 코드 리뷰와 잠재 버그 추론.
- 성능, 검색 품질, RAG 전략처럼 판단 비용이 큰 문제 분석.

### 둘 다 가능한 일

- 테스트 작성과 버그 수정.
- 기존 코드 읽기와 영향 범위 분석.
- `context-notes.md`에 의사결정 이유 기록.

권장 운영은 “Claude Code Opus가 큰 판단을 검토하고, Codex가 로컬 실행과 검증을 촘촘하게 맡는 방식”이다. 단, 파일 수가 적고 범위가 명확한 수정은 Codex가 바로 처리하고 Claude Code가 리뷰하는 편이 빠르다.

## 합의 프로토콜

1. 이슈가 생기면 담당 에이전트가 문제를 한 문장으로 정의한다.
2. 관련 파일, 로그, 테스트 실패를 실제 출력 기준으로 확인한다.
3. 수정 전 `docs/<feature>/context-notes.md`에 가설과 근거를 기록한다.
4. Codex와 Claude Code가 각각 짧은 수정안을 낸다.
5. 차이가 있으면 더 단순하고 검증 쉬운 안을 우선한다.
6. 합의된 안만 구현한다.
7. 구현 후 테스트 또는 최소 빌드 검증을 실행한다.
8. 결과와 남은 리스크를 `checklist.md`와 `context-notes.md`에 남긴다.

## 토큰 절약 전략

- Codex 쪽은 평소 답변을 짧게 유지하고, 긴 조사에는 `cavecrew-investigator`를 우선 고려한다.
- 단순 위치 탐색은 `rg` 결과 중심으로 요약한다.
- 긴 설계 논의는 문서에 누적하고, 다음 세션에서는 문서를 읽어 이어간다.
- `caveman lite` 수준의 압축 답변을 기본 후보로 둔다. 단, 신규 사용법 안내나 위험 작업 설명은 명확성을 우선한다.
- 이미 설치된 토큰 절약 스킬은 `caveman`, `cavecrew`, `caveman-review`, `caveman-commit`, `caveman-compress`, `caveman-stats`다.
- 현 시점에서 추가 설치가 꼭 필요한 토큰 절약 스킬은 확인되지 않았다.

## 작업 핸드오프 규칙

Codex에서 Claude Code로 넘길 때 남길 정보.

- 현재 목표.
- 바꾼 파일.
- 실행한 명령과 결과.
- 실패한 테스트나 미확인 항목.
- 다음에 해야 할 가장 작은 작업.

Claude Code에서 Codex로 넘길 때 남길 정보.

- 설계 결론.
- 구현 시 지켜야 할 경계.
- 확인해야 할 파일과 테스트.
- 합의가 필요한 열린 질문.

## 기능 작업 템플릿

새 기능은 아래 구조를 사용한다.

```text
docs/<feature-name>/
├─ plan.md
├─ checklist.md
└─ context-notes.md
```

각 문서의 역할.

- `plan.md`는 무엇을 왜 하는지 기록한다.
- `checklist.md`는 완료 여부를 체크한다.
- `context-notes.md`는 결정과 근거를 시간순으로 누적한다.

## 개발 착수 기준

다음 조건이 충족되면 구현을 시작한다.

- 성공 기준이 한 문장으로 정의됨.
- 영향 파일이 대략 식별됨.
- 테스트 또는 검증 명령이 정해짐.
- Codex와 Claude Code 중 구현 담당이 정해짐.
- 불확실한 결정은 `context-notes.md`에 남김.

## 완료 기준

- 변경 파일이 요청 범위에 맞음.
- 관련 테스트 또는 최소 검증이 실행됨.
- 실패 또는 미실행 검증이 명확히 보고됨.
- `checklist.md`가 갱신됨.
- 필요하면 하나의 논리 단위로 커밋됨.
